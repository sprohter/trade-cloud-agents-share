# 分享包维护规则

> 本文件定义 `agents-share/` 这份团队分享包的最小治理规则。

---

## 1. 目标

确保这份分享包长期保持：

- 可直接发给组员
- 可被本地 agent 直接读取
- 不混入个人数据和历史噪音
- 不逐步长成另一个个人工作区

---

## 2. 当前只保留两条主流程

当前分享包的一级主流程只保留：

1. 功能测试流程
2. 线上反馈问题处理流程

以下内容默认不作为平级主流程分享：

- 个人复盘
- 个人管理动作
- 历史迁移过程
- 个人本机调试习惯

---

## 3. 新内容放置规则

分享包内新增或维护 agent 相关文档时，默认只放在本分享包对应层级；如果组员把规则吸收到自己项目里，长期真源应放到本地 `.agents/` 框架内，不放仓库根目录。

| 内容类型 | 放哪里 |
|---------|------|
| 完整稳定的工作闭环 | `workflows/` |
| 主流程中某一步的处理方法 | `subflows/` |
| 系统操作、工具使用、配置说明 | `runbooks/` |
| 可直接复用的接入样板 | `config-templates/` |

判断口诀：

- 回答“整体怎么走” → `workflows/`
- 回答“这一步怎么做” → `subflows/`
- 回答“某系统 / 工具怎么操作或怎么配” → `runbooks/`

---

## 4. 默认禁止事项

- 把新细节直接升成新的主流程
- 把个人账号、密码、token 写入分享包
- 把个人绝对路径写成团队标准
- 把 Windows 本机路径、macOS home 路径或 Linux home 路径写成唯一入口；应使用 `<workspace-root>`、`<agents-root>`、`<user-home>` 或相对路径
- 把历史修复记录、迁移记录塞进主流程文档
- 在多个文件里重复定义同一条规则

---

## 5. 更新时的最小联动

新增或显著修改文件后，至少同步检查：

1. `../README.md`
2. `../routing.md`
3. 当前文件所在层级是否正确
4. 路径是否仍可跨 Windows / macOS / Linux 理解；无法兼容时是否明确 Windows 优先和降级说明

---

## 6. 版本策略

推荐把分享包当作“团队可发版资产”来维护：

- 大改结构时再单独打版本
- 日常只做小步更新
- 默认保持向后兼容，不频繁改目录层级

---

## 7. 推荐维护方式

- 先在个人真源里成熟
- 再裁剪进分享包
- 只同步“团队能直接复用”的部分
