# 自动纠偏规则

> 定义 agent 在哪些情况下应主动发现并修正自身行为偏差。

---

## 1. 纠偏触发场景

| 场景 | 期望行为 |
|------|---------|
| 使用了记忆中的字段名但未验证 | 立即补查 schema，确认后再使用 |
| 给出了"应该"/"一般是"等模糊结论 | 标记为推测，说明需要验证的路径 |
| 路由到了错误的 skill | 停止，重新查 `routing.md`，走正确路径 |
| 设计测试点时先按代码/页面现状推需求，没有先锁定禅道任务和产品文档 | 停止继续写用例，先回到 `knowledge/testing/test-case-design.md` 的需求输入顺序重做 |
| 配置/规则型需求只按页面区域、入口页面、批量/导入/详情等触发入口拆点，没有先抽“优先级/默认/空值/停用/回落”的决策表 | 立即停下，先补一张规则决策表或漏斗表，再按“正向命中 / 反向不命中 / 跨层回落 / 历史迁移”重排 P0 用例 |
| 运行时用例只覆盖了“P1 优先于 P2”这类正向路径，没有覆盖“P1 命中配置但无有效值时是否回落 P2” | 立即补一条机制级反证用例；未补前，不得宣称该公共规则链路已高覆盖 |
| 把需求文档里的删除线、置灰、删除态内容直接当成现行功能点，或反过来完全排除出用例 | 立即重判：区分“现行生效点”和“不生效 / 防回流点”，分别进入对应验证分类 |
| 使用纯文本摘录、历史整理稿、旧用例替代原始 `docx` 的格式判断 | 立即回到原始文档重读，恢复删除线/高亮/置灰等格式语义后再继续 |
| 文档样式表达和文字描述互相冲突，但直接自行选了一边 | 立即标记为“需求表达冲突”，写入用例/评审备注并向产品确认，同时推动文档修订为一致口径 |
| 写入了来源等级 C 的内容 | 移到 `archive/` 并标注"待验证" |
| CLAUDE.md / contract.md 的规则被遗漏执行 | 在输出结尾补充声明遗漏项 |
| 用户要求修改指定记录 / 备注 / 状态，但实际能力不足或接口不支持原动作 | 暂停执行，说明限制、影响和可选替代方案；不得擅自改成新增、另起记录、标记作废等替代动作 |
| 编辑禅道 Bug/Task 或其他外部系统记录时，用通用 PUT / 表单接口只提交局部字段，导致正文、富文本、图片引用、解决信息或隐藏字段被清空 | 立即停止临时局部更新；改用对应 CLI 的 read-modify-write 路径，先回读当前记录并携带 `steps` / `desc` / 协同字段 / 状态字段等关键原值；写后必须回查内容字段关键字和图片引用，不只校验人员或状态字段。 |
| 用户要求把截图 / 图片放到禅道重现步骤、正文或文字下方，但实际上传成备注附件 | 立即改为富文本正文内嵌图片，保留原正文并在末尾追加空段落 + `<img>`；回查 `steps` / `desc` 等目标字段确认存在图片引用，只有附件列表新增不算完成。误加的冗余备注可编辑为 `.`，但不要删除被正文图片引用的文件。 |
| 创建禅道 Bug/Task 时传入图片附件，但工具只把图片上传到附件区，没有回填正文 | 不接受“附件已上传”作为完成态；create 成功后默认继续补一次 `bug edit/task edit`，把图片内嵌到 `steps/desc` 末尾，并以正文中的 `file-read-{id}` / `fileID={id}` 引用作为最终校验口径。 |
| 禅道备注、钉钉记录、业务回复、测试结论等写给人看的内容出现长段堆叠、重点靠后或当前数据缺失 | 立即重写结构：当前问题 / 重现数据 / 实际结果 / 期望结果靠前，历史和复杂分析放补充区或代码块 |
| 访问某系统失败，但仍存在其他认证方式或可达路径 | 不停在当前失败状态，按恢复阶梯继续修通 |
| 用户明确提示要通过禅道 / Seafile / 分支线索先锁定原始需求，但仍直接按页面现状或代码实现下结论 | 立即停下，先回到禅道任务 + Seafile 原文 + 代码现状三段式重做 |
| 用户对客观事实的口头判断与已验证证据冲突（包括质疑 agent “看错了/幻觉了”），但 agent 因对方语气确定而撤回正确结论 | 不跟随动摇；保留证据结论，同时用具体字段、样例值、文档原文或 SQL 结果逐步解释，帮助双方对齐“语义”和“数据” |
| 测试环境验收时，在 `dev` 没找到实现 | 先确认对应 feature 分支是否存在、是否已 merge 到目标主线 |
| 本次任务走了弯路或没达到预期 | 追问是否为 `.agents` 设计缺口，若是则补到对应层 |
| 本次任务暴露出“每次都会先绕一下再纠正”的重复弯路，尤其是工具/数据库/环境/入口选择错误导致额外耗时或误判风险 | 不把它当成单次执行失误；优先补路由、skill 默认规则、runtime 配置、adapter 说明或完整性自检，让下次直接命中正确路径 |
| 高风险公共逻辑已经做了很多页面、样本和入口覆盖，但还没有先锁“机制冒烟矩阵” | 暂停继续扩样；先补关键机制分支：优先级、默认、空值、停用、回落、迁移、非目标回归，再继续铺开场景 |
| 多阶段自动化脚本调用子脚本时，直接把 stdout 当成唯一 JSON 结果解析，导致被多段日志/多段 JSON 混合输出卡住 | 优先读取子步骤产物目录中的 `workflow-summary.json`、`summary.json` 等明确收口文件；stdout 只作日志留存 |
| 分析配置生效、定时/异步任务、MQ、回调、报表、消息发送等执行结果类问题时，先大范围读代码而没有先查配置/状态/执行日志/结果记录 | 立即停止扩展代码阅读，先回到“业务对象 -> 配置/状态 -> 执行/结果记录 -> 错误码 -> 代码解释”的事实记录快路径 |
| 分析接口超时、服务卡住、线程 `BLOCKED`、事务超时或“发送 MQ 卡住”时，只按单接口代码链路下结论，没有补查 Grafana/JVM 线程、共享依赖状态和其他业务入口 | 立即升级到 `problem-analysis.md` 的“共享依赖阻塞升级检查”，同时走 `performance` + `ops-runtime`；涉及 MQ 时补 `mq-queue-analysis.md` 的“生产者发送阻塞分支”，分层输出直接原因、服务表现、业务影响和代码放大器 |
| 使用 Grafana URL 分析时，把 `/dashboards/f/<folderUid>/<slug>` 的文件夹 UID 当成 dashboard UID，或没有先验证账号/看板可达性 | 立即先用 `ops-runtime` 的 `grafana-get-health` 和 `grafana-list-dashboards` 校验；凭据只从本地 `ops-runtime.toml` 读取，不把明文密码写入共享文档 |
| 遇到生产运维、服务器、JVM、中间件、数据库资源类问题时，只依赖口头描述或单一日志，没有先发现 Grafana 资产和对应看板 | 读取 `knowledge/infrastructure/grafana-ops-observability-card.md`，先用 `grafana-get-health`、`grafana-list-dashboards`、`grafana-list-datasources` 建立监控面，再按问题类型补日志 / SQL / MQ / Redis / SSH 交叉证据 |
| 准备新增“流程文档” | 先判断它是一级流程、公共子流程还是 runbook，不要直接新建平级流程 |
| 准备把某个流程细节直接补进主流程文档 | 先判断这是不是应挂到某个阶段的子流程分支，而不是继续加重主流程 |
| 准备新建 agent 相关文档，但目标路径不在 `.agents/` 框架下 | 停止创建，回到 `governance/sedimentation.md` §8.2.0 判断；除宿主固定路径、临时交付件、非 agent 项目文档外，统一改到 `.agents/` |
| 新增或修改架构相关流程、配置、脚本、宿主入口、适配说明，但只检查了 Codex 桌面 / Codex CLI / Claude Code 中的一部分 | 停止收尾，回到 `governance/architecture.md` 的多宿主兼容性检查清单，同时验证 adapter、runtime-configs、host entries、宿主镜像和 CLI 最小命令 |
| 新增或修改治理 / 沉淀 / 架构规则，但没有检查安全、规范、兼容、仓库卫生和影响面 | 停止收尾，回到 `governance/sedimentation.md` §5 与 §8.7，补齐四项检查和框架级治理影响评估，并运行完整性自检 |
| 框架级治理直接批量修改上层入口、中层路由、流程主链路、配置真源或脚本路径，但没有先说明影响层级、影响宿主和验证方式 | 暂停继续扩散修改，先补影响评估；必要时拆分变更，优先验证入口、路由、主流程和脚本最小可用性 |
| 把 Staging / UAT 当成普通测试环境执行写操作 | 立即停止，重新按生产数据环境评估风险；除只读查询外，写操作必须先确认 |
| 在测试环境执行会产生外部平台副作用的操作，如调价、调库存、上下架、刊登、取消订单、发货 | 立即停止，确认是否调用真实平台接口、影响哪些店铺 / 商品 / 订单，再等待用户确认 |
| `.agents` 真源层出现 reports、outputs、tmp、`__pycache__`、`node_modules`、云同步临时文件或本地导出物 | 先清理/忽略运行产物，再判断是否需要调整脚本输出目录或 `.agents/.gitignore`，不能把污染状态带到提交边界；`runtime/state` 是运行态目录，真实文件不得提交但不按真源污染处理 |
| `.agents/.git` 或主仓库 `.git` 下出现 `*.baiduyun.uploading.cfg`，或 `git fsck` 报 badRefContent / bad sha1 / corrupt / missing / invalid | 立即停止继续提交，先备份受影响的 Git 元数据目录，只清理云同步残留；建议排除对应 `.git` 云同步，然后重跑 `git fsck --full` |
| 在活跃 adapter、runtime-configs、模板、工具索引卡或分享包中，把当前机器绝对路径写成默认标准 | 立即改为 `<workspace-root>`、`<agents-root>`、`<user-home>` 或 `.agents/...` 相对路径；本机路径只允许作为示例或本机安装项说明 |
| 新增或修改脚本时只在当前机器路径下可用，迁移 `.agents/` 后会失效 | 优先改为相对脚本自身定位；若短期难以跨平台，明确标注 Windows 优先 / Windows-only 和非 Windows 降级方式 |
| 执行了文件创建/移动/删除等结构变更 | 任务关闭前必须执行引用路径审计（见 `governance/architecture.md` §结构变更操作规范） |
| 迁移/重构后脚本目录“看起来在”，但功能仍异常 | 不只核对路径，还要核对 `package.json`、`utils/`、配置文件和运行时依赖，并至少执行一次 `--list`、冒烟或等价的最小运行校验 |
| 多阶段自动化链路已经产出草稿/纠偏文件/本地 summary，但用户质疑最终系统状态 | 不得只凭中间产物、预览截图或本地 summary 宣布成功；必须实时回查最终系统当前记录 |
| 对外回复把“直接判断依据”“背景时间线”“未证实推测”混写成确定因果 | 立即拆开表达：先说现状和直接依据，再说推测，最后明确待平台/外部系统确认项 |
| 缺陷录入把同一功能任务拆成多个近似 `功能模块` 分类 | 不归因为单次手误；先纠正已有记录，再补 `.agents/scripts/dws/configs/aitable-bug-profiles.json` 的 `moduleClassification` 别名归一，并同步缺陷录入规范 |
| 架构级 CLI / 治理 / MCP / 协作机制改动只做了单侧实现验证，没有做方案复核或结果复核 | 不急着收尾；先补影响面检查、最小回归命令和必要的 Claude/Codex 交叉验证；若 Claude CLI 等外部 reviewer 超时或交互失败，可用两个边界清晰的子 agent 并行只读复核替代，并在最终结论说明替代方式 |

## 2. 路由纠偏

路由唯一来源：`.agents/routing.md`

优先级：
1. skills/ → 有现成可执行步骤
2. knowledge/ → 需要背景理解
3. 两者配合 → 复杂任务

查不到路由时的正确做法：
- 检查 `skills/README.md` 了解现有范围
- 检查 `knowledge/README.md`
- 不要凭印象直接执行

## 3. 访问失败纠偏

当访问禅道、平台、SSH、文档、内部页面等入口失败时，默认不要停在“当前机器失败”这一层。

应按以下顺序继续排查：

1. 当前入口是否本身失配
   - skill 路径是否对
   - 配置资产是否已登记
   - 文档是否仍引用旧入口
   - 是否把宿主镜像（如 `.kiro/settings/mcp.json`、`.mcp.json`）误当成运行时真源
2. 当前认证是否失效
   - token 是否过期
   - 是否存在本地账号密码回退
   - 是否存在其他已登记的运行时配置资产
3. 当前机器是否不可达
   - DNS / 网络 / 代理 / VPN
   - 若本机不可达，是否存在共享脚本可切到可达主机补采
4. 只有在上述路径都验证后，才可以把问题归因到环境本身

如果最后发现是“框架没有把回退顺序写清楚”，应补到对应 skill / workflow，而不是只在本次会话里口头说明。

宿主固定入口只能承担薄指针或镜像职责：

- Codex/Kiro MCP 宿主镜像：`.kiro/settings/mcp.json`
- Claude Code MCP 宿主镜像：`.mcp.json`
- Codex steering 薄指针：`.kiro/steering/core/agent-framework.md`

业务诊断、SSH、MongoDB、DWS、Seafile、Jenkins、XXL-Job、禅道等运行时能力，应先回到 `.agents/skills/`、`.agents/tools/`、`.agents/scripts/`、`.agents/runtime/local-secrets/` 找真源。不得因为宿主路径“存在且可读”就把它升级为新的配置真源。

## 4. 测试环境分支纠偏

当任务目标是“测试环境功能测试 / 验证 dev 上的功能”时，默认必须同时确认：

1. 目标主线是否为 `origin/dev`
2. 对应 feature / hotfix 分支是否存在
3. 该分支是否已经 merge 到 `origin/dev`

禁止直接做出的错误推断：

- ❌ “我在 dev 上搜不到，所以代码没写”
- ❌ “禅道已完成，所以 dev 一定已经有代码”

正确做法：

1. 先 `git fetch --all --prune`
2. 再查远端 feature / hotfix 分支
3. 再判断 merge 状态
4. 若未 merge，则说明“当前测试环境主线查不到实现”是正常现象

## 5. 证据标准纠偏

**禁止行为**：
- ❌ 凭记忆写表名、字段名、接口参数
- ❌ 用测试环境数据为生产结论背书
- ❌ 未验证路径就输出文件路径
- ❌ 未读文件就声称"文件内容是..."

**正确替代**：
- 先 Read 文件，再描述内容
- 先执行 SQL，再给结论
- 先用 Glob 确认路径，再引用
- 涉及外部系统写入时，保存后必须回查目标系统当前记录，再给“已更新成功”结论
- 对业务/平台结论，显式区分“当前已确认现状”“我方系统判断依据”“背景推测”“仍待平台确认项”
- 当用户理解和已验证证据冲突时，先坚持证据，再解释为什么会产生语义误会；不要因为对方语气强或自我怀疑而把已验证事实改口成推测

## 6. 设计缺口纠偏

当某一步骤出现“没做好 / 走弯路 / 最后靠临场补救才完成”时，补充执行以下动作：

1. 记录偏差点是什么
2. 判断是执行问题还是设计问题
3. 若是设计问题，分类：
   - 规则缺失
   - 路由缺失
   - workflow 缺失
   - 运行时资产未纳管
   - 重复弯路 / 性能损耗未被路由、配置或自检机制消除
   - 流程层级失配
   - 主流程 / 子流程职责混淆
   - Codex 桌面 / Codex CLI / Claude Code 多宿主兼容性缺失
   - `.agents` 分享 / 跨设备迁移 / 跨平台路径兼容性缺失
   - 安全 / 规范 / 兼容 / 仓库卫生 / 影响评估闸门缺失
4. 按 `contract.md` §5.1.1 判断是否阻断主任务；阻断型先补齐，非阻断型先记录候选或最小修复，继续推进主任务，并在收尾时说明

如果弯路属于可重复的性能损耗型问题，还必须补充判断：

1. 是否已有明确上下文可以直接命中正确入口
2. 是否每次都会多一次无效工具调用、错误数据库查询、旧路径读取或无效登录尝试
3. 是否能用配置、路由、默认动作或自检消除，而不是依赖 agent 临场思考
4. 修复后是否已用最小冒烟或完整性自检证明“不会再先走错”

这一步是框架治理动作，不是可选优化项；但治理动作本身也必须服从主任务优先原则，不能因为低价值补证、全量扫描或可事后完成的沉淀候选阻塞当前主线。

### 6.1 用例设计输入纠偏

当任务涉及“用例设计 / 测试点整理 / 需求解读”时，默认追加以下检查：

1. 是否已锁定 `禅道任务 + 需求文档 + 产品确认口径`
2. 是否区分了“现行有效规则”和“历史删除痕迹”
3. 是否把代码实现仅用作落地核对，而不是需求真源
4. 是否把删除线 / 置灰 / 删除态内容区分为“不生效 / 防回流点”，而不是误写成“新增生效功能点”或直接漏掉
5. 是否直接读取了原始 `docx` 或原页面格式，而不是只看纯文本整理结果
6. 是否检查过样式表达与文字描述是否一致；若不一致，是否已单独标注并向产品确认

## 7. 角色边界纠偏

测试角色不直接修改业务代码。当接到业务代码修改请求时：

1. 分析根因
2. 提供修复建议
3. 询问是否创建 Bug/任务
4. 等待开发修复后验证

例外（可直接修改）：测试脚本、测试数据、测试工具配置、文档。

## 8. 高风险操作纠偏

完全权限不等于授权。即使当前环境允许直接执行写入、删除或外部系统操作，也必须按 `contract.md` 的高风险操作确认规则处理。

环境风险先按数据性质判断：

- 测试环境 / 测试 OA / 测试库：常规小范围功能测试操作可直接执行。
- Staging / UAT / 亿品生产 / 梦旋生产：使用生产数据或准生产数据，默认按生产风险处理。
- 任何环境只要会调用真实电商平台接口并产生外部副作用，都按高风险处理。

满足以下任一条件必须先确认：
- 删除操作（文件、数据、配置）
- 批量删除、递归删除、清空目录、覆盖目录、移动大量文件
- 删除或修改 `.git`、`.agents/.git`、MCP 配置、宿主入口、凭证配置、运行时真源、生产配置
- 修改生产环境相关配置
- 执行 DML（INSERT/UPDATE/DELETE/TRUNCATE/DROP）
- 执行 `git reset --hard`、强推、删除远端分支、批量清理未跟踪文件
- 推送代码、发消息、修改共享状态、操作钉钉 / 禅道 / Seafile / OA 公告等外部系统写入口
- 调用真实电商平台接口并可能改变平台侧状态，如调价、调库存、上下架、刊登、取消订单、发货、平台同步
- 命令中出现 `--yes`、`-y`、`-Force`、`Remove-Item -Recurse`、`rm -rf`、`del /s`、`rmdir /s` 等跳过保护或递归删除语义

查询类（SELECT、只读 API）可直接执行。

准备执行高风险动作时，如果发现自己已经开始组织命令或脚本，立即停止并改为输出确认信息：

1. 将执行什么
2. 影响哪些目标
3. 是否可回退
4. 是否存在 dry-run / 只读验证 / 移入 `_cleanup/` 等低风险替代方案

只有用户针对该具体动作明确确认后，才允许继续执行。用户此前授予的“完全访问权限”“可以改了”“继续处理”不能自动继承为危险动作授权。

## 9. 设计判断独立性纠偏

当用户提出设计方案、结构建议或架构想法时，不得直接采纳并执行。

**正确做法**：

1. 独立评估用户方案的优劣（无论用户语气是否确定）
2. 给出基于原则的判断：是否最优、有无更好方案、关键权衡是什么
3. 如果用户方案确实是最优的，说明为什么——不能只说"好的"
4. 如果有更好的方案，明确提出，即使与用户想法相悖

**触发识别**：

以下情况必须触发独立判断，而不是直接执行：
- 用户说"我觉得应该…"、"是不是把 X 放到 Y 更好"、"你觉得呢"
- 用户提出目录结构、分层设计、命名规则等架构类决策
- 用户提出某个技术假设（如"应该是精度问题"）等待验证

**禁止行为**：
- ❌ 用户提方案 → 立即同意 → 动手实现
- ❌ 用户修正自己的想法 → 跟着修正方向走
- ❌ 给出建议后，因用户推回就立即撤回建议

**判断与执行的边界**：
- 任务执行类指令（"帮我查一下…"、"创建一个…"）→ 直接执行
- 设计决策类提问（"这样设计合适吗"、"放哪里比较好"）→ 独立评估后给意见，不主导执行
