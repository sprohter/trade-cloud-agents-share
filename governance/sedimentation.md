# 沉淀规则

> 定义哪些内容值得沉淀、如何分类、如何写入。

---

## 1. 沉淀触发条件

满足以下任一条件，**且通过 §2 否定清单过滤后**，才在当次会话结束前主动输出沉淀内容：

- 走通了一个新的完整操作流程（如 Playwright 配置页验证闭环、功能测试流程、bug 验证流程）
- 发现某个字段/接口的真实语义与直觉不同，且**不读文档就会踩坑**
- 发现某个 agent 行为偏差的根因，且**现有规则未覆盖**
- 任务过程中出现走弯路、被纠正、缺少标准步骤的情况 → 识别为**框架缺口**，写入位置按 `routing.md` 路由，不在任务结论里硬编码
- 任务过程中出现**可重复的弯路**：例如每次都会先查错库、先走错工具、先读错入口、先尝试已知不可用路径，再靠临场思考纠正；且该弯路会明显增加耗时、工具调用、token 成本或误判风险 → 默认识别为**高价值设计缺口**，优先补路由、配置、默认动作或完整性自检
- 用户明确指出“测试点/用例不是基于需求文档设计”“把删除线/置灰/删除态内容当成现行需求”“按代码实现反推需求”等原则性偏差 → 默认视为**高优先级框架缺口**，必须判断是否同步补 `knowledge/` 真源和 `governance/self-correction.md`
- 如果同一轮里连续出现 2 次及以上“这条应该沉淀，但现有沉淀规则没有主动提醒到”的情况 → 不只补业务真源，还要**反向补 `governance/sedimentation.md` 或 `governance/self-correction.md`**
- 当本轮已经补了 `skill/knowledge/governance` 真源后，如果用户继续追问“为什么这条之前没沉淀 / 规则为什么没覆盖”，默认进入**规则层复盘**，检查是否要继续补治理真源
- 如果一条内容已经在会话中被推动并最终落地到 `skill/knowledge/governance` 真源，说明它**至少通过了当次执行判断并被实际写入**；但“已落地”**不自动等于**“高价值”。后续仍应回看：它到底是高复用真源，还是一次偏重当场推进的低价值记录
- 已在会话中直接落地为文件（skill/knowledge/governance），且 routing + asset-registry 已更新 → 不再重复沉淀，直接声明"已落地"

> **核心判断**：沉淀的价值 = 下次遇到同类任务时，不读这条就会走弯路或出错。大模型凭常识能推理出的通用原则不需要沉淀。

### 1.1 “价值”具象化判断

当“这条到底值不值得沉淀”比较模糊时，优先看下面 5 个维度：

1. **复发概率**：下次还会不会再遇到，而不是只在这一次成立
2. **误判代价**：如果不沉淀，下次是否容易得出错误结论、走弯路、浪费大量时间
3. **证据门槛**：这条结论是否依赖代码、SQL、原始文档、系统特定规则，不能只靠常识猜出来
4. **决策影响**：它会不会改变后续怎么查、怎么判断、怎么回复、怎么选 skill / knowledge / workflow
5. **跨任务复用性**：它是只服务当前对象，还是能迁移到同类任务、同类平台、同类协作场景

判断建议：

- 满足 3 项及以上，通常可视为**高价值内容**
- 只满足 1-2 项，通常更像**低价值或局部记录**
- “已经落地”只能说明本轮执行时认为它值得写，**不能替代这 5 项价值判断**

### 1.2 价值评级

当 agent 判断存在沉淀候选时，默认按以下 3 档给出评级，而不是只做“沉淀 / 不沉淀”二元判断：

| 级别 | 含义 | 默认动作 |
|------|------|---------|
| 高价值 | 明显会改变下次同类任务的速度、准确性、稳定性或复用性 | 优先建议沉淀；若已确认来源等级 A/B，可直接落到真源 |
| 中价值 | 有一定复用性，但边界、适用范围或收益还不够稳定 | 先整理成候选，给出我的判断和适用边界，再由用户确认是否落真源 |
| 低价值 | 更像一次性记录、局部现场备注、弱复用经验 | 默认不直接落真源，但仍要在“沉淀候选”里列出来，避免遗漏，只是需要额外确认 |

补充口径：

- **低价值 ≠ 不记录**
- 低价值或不确定项，仍应按沉淀规则整理成候选，让用户看到，而不是完全依赖用户自己发现
- agent 应明确给出自己的判断，不要把“值不值得沉淀”全部推回给用户

### 1.3 重复弯路 / 性能损耗型缺口

以下情况不按“一次性经验”处理，而按**高价值设计缺口**处理：

| 判断项 | 说明 |
|--------|------|
| 已知上下文明确 | 用户、任务、环境或配置已经给出足够线索，理论上应直接命中正确入口 |
| 每次或高概率复发 | 不是偶发失误，而是后续同类任务仍会重复绕路 |
| 代价明显 | 会多查一次库、多启动错误工具、多读旧文档、多跑无效脚本，造成耗时、token、工具调用或误判风险 |
| 可机制化修复 | 能通过 `routing.md`、skill 规则、runtime-configs、MCP 配置、默认动作、完整性自检或脚本校验消除 |
| 可验证 | 修复后能用自检、冒烟、grep、最小命令证明不再绕路 |

默认处理顺序：

1. 先修当前任务的真实入口或配置，避免继续绕路
2. 再判断缺口属于路由、skill、knowledge、runtime、adapter 还是 governance
3. 能补自检的，优先补自检；因为自检能防止“几份配置彼此一致但一起错”
4. 如果只写文档仍容易被跳过，应补默认动作或脚本校验，而不是只补说明
5. 收尾时明确说明：这次消除的是哪条重复弯路，以及用什么验证证明已消除

### 1.4 沉淀执行效率要求

沉淀任务默认也要控制工具成本和验证噪音：

1. **先定位，再读取**：已知道目标文件或章节时，优先用 `rg -n`、`Select-String`、行号片段读取；不要先整篇读取大型 runbook / JSON / 日志。
2. **先查活跃真源，再查 archive**：archive 用于“写入前查重”和历史补证，不作为第一轮大范围全文扫描入口；关键词搜索应限制路径和文件类型。
3. **先用任务已知 ID 缩小范围**：已知禅道父任务 / 子任务 ID 时，先 `task_get` 读取父任务 `children` 或指定子任务，不先拉全量 `task_list`。
4. **按真实仓库边界自检**：如果修改对象位于独立仓库 / 嵌套 `.git`（如 `.agents/.git`），用该目录自己的 git 状态和 diff 自检；不要用外层工作区 diff 判断是否改到。
5. **自检命令要验证关键行为**：脚本变更除语法检查外，优先补一个最小函数级 / dry-run 验证，证明本次要消除的弯路确实被消除。

## 2. 否定清单（不该沉淀什么）

以下内容**不应沉淀**，即使看起来有价值：

| 类型 | 为什么不沉淀 | 举例 |
|------|-------------|------|
| 通用设计原则 | 大模型自带推理能力，不需要记忆 | "调用前要声明超时"、"出错要降级"、"禁止无声等待" |
| 小场景工具修复 | 一次性问题，下次不会重复踩 | Read 工具 pages 参数不能传空字符串 |
| 大模型自有能力 | 不写也不会忘 | "超时了要报错"、"参数错了要修正" |
| 已落地文件的复述 | 文件本身已在 routing 中可命中 | 迁移方案已写入 codex-mcp-fault-tolerance.md，不需要在 lessons.md 再记一条 |
| 通用编程常识 | 不属于项目特有知识 | "SQL 要防注入"、"catch 要处理异常" |

**判断口诀**：问自己——"这条如果我不写，下次大模型会不会自己想到？"会 → 不沉淀；不会 → 才沉淀。

## 3. 内容分类

| 类别 | 放哪里 | 示例 |
|------|--------|------|
| 操作技能（怎么做） | `skills/` | layui 下拉的正确点击方式 |
| 业务口径（背景是什么） | `knowledge/business/` | 某字段在不同平台的含义差异 |
| 技术模式（项目级经验） | `knowledge/testing/` 或 `knowledge/infrastructure/` | Playwright 脚本基础设施 |
| 治理规则（如何运作） | `governance/` | 本文件 |
| Agent 适配（接入说明） | `<agent-name>/adapter.md`（如 `claude-code/adapter.md`） | Claude Code 的跨会话注意事项 |

### 3.1 M1-M4 记忆分层口径

新增或修改 agent 记忆类内容时，先判定它属于哪一层，再决定写入位置。当前阶段只采用分层口径，不新增活跃 `memory/` 目录。

| 层级 | 作用 | 当前写入位置 | 默认读取方式 |
|------|------|--------------|--------------|
| M1：索引记忆 | 帮 agent 快速知道该去哪里读 | `README.md`、`routing.md`、`knowledge/*/README.md`、`governance/asset-registry.md` | 起手优先读取，正文保持短 |
| M2：事实记忆 | 保存稳定业务事实、字段口径、环境口径、团队规则 | `knowledge/business/`、`knowledge/infrastructure/`、相关规则卡 | 命中领域后读取，必须标注来源和边界 |
| M3：流程记忆 | 保存可重复执行的 SOP、runbook、验证流程、工具步骤 | `skills/`、`knowledge/testing/*runbook*`、`tools/` | 进入执行阶段读取，强调输入、步骤、风险、输出和验证 |
| M4：归档记忆 | 保存历史案例、事故现场、过期方案、完整证据 | `case-studies/`、`archive/`、`governance/migration-log.md` | 默认不全量读，只在相似任务或追溯时查 |

写入判断：

1. 只负责导航或摘要的内容写 M1，不承载长正文。
2. 稳定事实写 M2，不混入长执行流程。
3. 可重复操作步骤写 M3，不长期堆一次性现场。
4. 历史现场、事故复盘、过期方案写 M4，不提升为默认上下文。
5. 同一内容只能有一个主归属；其他层只做索引或摘要，避免重复维护。

## 4. 来源等级

| 等级 | 说明 | 是否可直接写入 |
|------|------|--------------|
| A | 代码位置 + SQL结果 + 截图 + 文档原文 | ✅ 直接写入 |
| B | 可复现的操作步骤 + 结果验证 | ✅ 直接写入 |
| C | 口头印象、未核实推断 | ❌ 先放 `archive/`，标注"待验证" |

## 5. 写入前检查清单

- [ ] 来源等级是 A 或 B
- [ ] 不与现有内容重复（先检索）
- [ ] 表达对"下次任务"有用，不是一次性内容
- [ ] 文件路径指向实际存在的文件
- [ ] 通过 §2 否定清单过滤（不是通用原则、不是小场景修复、不是大模型自有能力）
- [ ] 沉淀价值够高：不写这条，下次大模型会不会自己想到？会 → 不沉淀
- [ ] 如果是“流程类内容”，已先判断它属于一级流程 / 公共子流程 / runbook，而不是凭感觉新建平级流程文档
- [ ] 如果是“流程细节”，已先判断它应挂到哪个一级流程、哪个阶段，优先写成该阶段的子流程分支
- [ ] 如果是“记忆类内容”，已按 §3.1 判断主归属是 M1 索引、M2 事实、M3 流程还是 M4 归档，并避免一份内容多处重复维护
- [ ] 如果本轮是“被用户纠正回正轨”的原则性问题，已判断这是不是要同时补 `knowledge/` 与 `governance/self-correction.md`
- [ ] 如果本轮出现“每次都会先绕一下再纠正”的重复弯路，已判断是否应补路由、配置、默认动作或完整性自检，而不是只把它当成一次性经验
- [ ] 如果本轮已补了真源，但用户继续指出“沉淀规则本身没覆盖到”，已反查是否还要补 `governance/sedimentation.md`
- [ ] 如果本轮只是补现有真源的局部规则，已判断是否真的需要再写 `migration-log.md`，避免把所有小改动都扩成治理记录
- [ ] 如果准备新建 agent 相关文档，已确认路径位于 `.agents/` 框架内；若不是，必须先说明例外理由并确认它不会成为 agent 真源
- [ ] 如果准备新建或重写 agent 相关文档，正文、流程说明、注意事项和输出模板默认使用简体中文；英文只保留代码标识、命令、路径、API 名、表名、字段名、外部产品名或外部原文标题
- [ ] 如果准备新建或修改架构相关流程、配置、脚本、宿主入口、适配说明，已同步检查 Codex 桌面、Codex CLI、Claude Code 的兼容性，避免只对单一宿主生效
- [ ] 如果新增或修改的内容会随 `.agents/` 分享给其他设备或其他 agent，已避免把当前机器绝对路径写成标准；路径优先使用 `.agents/...`、`<agents-root>`、`<workspace-root>`、`<user-home>`
- [ ] 如果准备修改治理、沉淀、架构、脚本或运行时配置，已同步检查安全边界、命名/放置规范、多宿主兼容性和仓库卫生，不把运行产物、缓存、云同步临时文件或凭证写入框架真源
- [ ] 如果准备做框架级治理，尤其涉及上层入口、中层路由、流程主链路、配置真源或脚本路径，已先完成影响评估，再决定是否调整

## 6. Claude Code 专属适配

Claude Code 没有跨会话的候选池，沉淀必须在当次会话结束前完成。

**Claude Code 专属写入目标**（补充 §3 通用分类表）：

| 内容类型 | 写入路径 | 收录标准 |
|---------|---------|---------|
| Agent 行为偏差纠正 | `claude-code/lessons.md` | **仅限 Claude Code 自身行为错误**：跳过规则、越权执行、错误判断、被纠正的推断等。不收录：过程发现、外部环境状态、一次性事件记录 |

**lessons.md 收录前置过滤**（输出沉淀建议前必须确认）：

- [ ] 这是 Claude Code 做了不该做的事，还是任务中发现了意外的外部状态？→ 是后者则不收录
- [ ] 下次遇到同类任务时，这条能改变 Claude Code 的行为决策吗？→ 不能则不收录
- [ ] 已有文档（migration-log、README、knowledge/）覆盖了这个发现吗？→ 已覆盖则不重复

**输出格式（当触发条件满足时）**：

```
## 本次沉淀建议

### [技能/知识/治理/适配] 类别

**标题**：xxx
**价值评级**：高 / 中 / 低
**来源**：task-XXXXX，2026-XX-XX
**判断依据**：复发概率 / 误判代价 / 证据门槛 / 决策影响 / 跨任务复用性 中命中的关键点
**内容**：...
**写入路径**：.agents/knowledge/testing/automation.md
```

如果同轮存在多个候选，默认至少分成两组输出：

1. **建议直接沉淀**
2. **低价值 / 不确定，待确认是否沉淀**

输出后询问用户是否确认写入，确认后执行 Write 操作。

## 7. 沉淀质量原则

- 宁可少写，不要写错：等级 C 的内容绝对不能直接落库
- 一条沉淀解决一个问题，不要把多个陷阱塞进同一条
- 示例代码优先于文字描述
- 写完后在文件末尾注明来源任务和日期

## 8. 文档创建规则（迁自 doc-creation-rules.md）

> 本节原为独立文件 `governance/doc-creation-rules.md`，2026-04-20 合并至此。

### 8.1 创建前必做三步

新增文档前，必须先写出这三项：

1. 这份文档解决什么动作
2. 这份文档属于哪一层
3. 这份文档为什么不能直接复用已有文档

如果这三项写不出来，就先不要创建新文档。

### 8.2 放置规则

#### 8.2.0 根路径准入规则

**所有 agent 相关沉淀文档默认只能创建在 `.agents/` 框架下。**

这里的“agent 相关文档”包括：

- skill、knowledge、governance、template、prompt、tool index、case study、share package、collab 记录
- agent 适配说明、运行时配置说明、自检报告、框架修复记录
- 测试流程、线上问题处理流程、工具使用规则、沉淀规则、协作规则

默认禁止新建或继续维护为真源的位置：

| 禁止位置 | 正确做法 |
|----------|----------|
| 仓库根目录 | 放到 `.agents/README.md`、`.agents/templates/` 或对应子目录 |
| `docs/codex/` | 放到 `.agents/knowledge/`、`.agents/governance/`、`.agents/case-studies/` 或 `.agents/archive/` |
| `.kiro/docs/`、`.kiro/templates/` | 仅保留 tombstone / 宿主兼容说明；真源放 `.agents/` |
| `.codex/`、`.claude/`、`.agent/` | 仅保留宿主配置或本地运行态说明；通用知识回收到 `.agents/` |
| `scripts/` | 脚本本体放 `.agents/scripts/`，索引卡放 `.agents/tools/` |

允许例外，但必须显式说明原因：

1. 宿主固定读取路径，如 `AGENTS.md`、`CLAUDE.md`、`.mcp.json`、`.kiro/settings/mcp.json`
2. 非 agent 沉淀的业务项目文档，如产品 README、接口说明、业务代码模块文档
3. 用户明确要求输出到桌面或临时目录的交付件；若后续要作为长期规则复用，必须再回收进 `.agents/`
4. 第三方工具必须放在固定目录的配置文件；这种文件只能做镜像或薄指针，不能承载通用知识真源

如果不确定是不是 agent 相关文档，默认按“是”处理，先放 `.agents/`。

| 路径 | 放什么 | 不放什么 |
|------|--------|---------|
| `skills/{skill-name}/` | SKILL.md、README.md、与该 skill 强绑定的子文档 | 业务知识、临时分析 |
| `knowledge/{domain}/` | agent 执行任务时需要读取的背景知识 | agent 执行操作的步骤（→ skills） |
| `governance/` | 框架自身的治理规则 | 业务内容 |
| `tools/` | 可执行脚本的索引卡（脚本在 scripts/） | 脚本本体 |
| `templates/` | 操作模板、记录模板、评估模板 | 业务知识 |
| `case-studies/` | 任务专属测试执行档案 | 通用规则 |
| `archive/` | 已退役内容 | 当前活跃资产 |

判断：agent 执行任务时需要参考 → `knowledge/`；agent 执行操作步骤 → `skills/`

### 8.2.1 流程文档分层规则

测试相关流程文档默认按 3 层理解：

| 层级 | 定义 | 当前测试领域真源 |
|------|------|------|
| 一级业务流程 | 对应日常反复发生的完整工作闭环 | `knowledge/testing/methods.md` |
| 公共子流程 | 多个一级流程都会复用的某一步 | `skills/workflows/`、`knowledge/testing/fix-verification-pass-standard.md` 等 |
| Runbook / 工具手册 | 某个平台、系统、脚本的执行说明 | `knowledge/testing/*-runbook.md`、`tools/` |

补充设计原则：**一级流程与子流程分开设计，子流程作为一级流程某个阶段的分支挂载**。

| 层级 | 推荐设计方式 | 典型写法 |
|------|------|------|
| 一级业务流程 | 只保留稳定主链路 | 触发条件、关键阶段、终点产出 |
| 子流程 | 作为某阶段的分支文档 | “走到问题分析阶段后，功能问题怎么查” |
| Runbook | 作为子流程里的工具说明 | “AI 表格怎么维护状态”“公告页面怎么操作” |

当前测试领域默认只保留 4 条一级业务流程：

1. 功能测试流程
2. 线上反馈问题处理流程
3. 项目周计划测试任务评审
4. 项目重大问题月度复盘

新增“流程文档”前必须先判断：

- 能否挂到上述 4 条一级流程之一？
- 能否挂到该一级流程的某个阶段，写成子流程分支？
- 如果只是共享步骤，是否应写成子流程？
- 如果只是平台/脚本操作，是否应写成 runbook？

只有同时满足“独立触发、独立终点、反复发生”时，才允许提议新增一级流程；并且必须同步更新 `methods.md` 和 `routing.md`，不能只单独落一个新流程文件。

默认禁止的错误沉淀方式：

1. 主流程里直接塞大量代码定位、SQL 判断、系统点击细节
2. 某个专项分析刚积累一点内容，就直接升成新的一级流程
3. 把“聊天入口”“AI 表格维护”“公告维护”这类节点误写成平级流程

### 8.3 命名规则

- 一律 `kebab-case`
- 文件名表达动作或主题，不写空泛标题
- 不使用 `new-` `final-` `temp-2` 这类一次性命名
- 推荐后缀：`-guide.md`、`-runbook.md`、`-checklist.md`、`-overview.md`

### 8.4 链接与索引规则

- 已存在文件必须写成真实可点击链接
- 未创建文件不能伪装成可点击链接
- 真源规则：如果一份文档是某类规则的唯一真源，上级文档只做引用，不在其他文档重复展开

### 8.5 内容模板

所有新文档最少包含：目的 → 适用场景 → 正文 → 使用边界。

三类文档的建议结构：

- **治理文档**：目标 → 当前问题 → 规则或方案 → 执行口径 → 后续演进
- **Guide/Runbook**：目的 → 输入条件 → 执行步骤 → 常见分支 → 输出格式
- **Reference**：说明 → 数据表/映射表/列表 → 使用注意事项

### 8.6 创建时的禁止事项

1. 还没确认层级就直接写进正式目录
2. 把迁移历史写进 skill 入口
3. 同一规则在多处重复定义
4. 因为"感觉以后可能有用"就直接建新 skill
5. 写出看起来能点、实际无法解析的链接

### 8.7 联动更新清单（文件变动后必查）

**新建文件**：asset-registry 登记 → README 补行 → routing 补触发词（如是 skill/knowledge）→ migration-log 记录

**架构相关流程 / 配置 / 脚本 / 宿主入口变更**：除普通索引更新外，必须同步执行多宿主兼容性检查。至少确认以下内容：

1. Codex 桌面、Codex CLI、Claude Code 的 adapter / 入口说明是否仍指向同一 `.agents/` 真源
2. 各宿主 runtime-configs 是否登记了新增或变更的配置资产
3. `.agents/host-entries/` 模板与根目录 `AGENTS.md` / `CLAUDE.md` 生成结果是否一致
4. 如涉及 MCP、脚本、凭证、运行时配置，是否同步处理 `.agents/runtime/` 真源与宿主固定镜像
5. 如涉及命令行脚本、环境变量、cwd、模型 / provider、MCP server，是否单独确认 Codex CLI 可用，不能用 Codex 桌面通过来替代
6. 如涉及路径、模板或分享包，是否仍能在 Windows / macOS / Linux 上按相同 `.agents/` 真源理解；无法兼容时是否明确 Windows 优先和降级方式

**治理 / 沉淀 / 架构规则变更**：必须额外执行安全、规范、兼容、仓库卫生四项检查。

1. 安全：是否涉及凭证、生产写操作、外部发送、删除或不可逆动作；敏感信息只能登记读取规则，不能复制正文
2. 规范：新内容是否位于 `.agents/` 正确层级，是否已有同主题真源可合并，是否更新 routing / README / asset-registry
3. 兼容：Codex 桌面、Codex CLI、Claude Code 是否都能按同一路由读取；宿主固定入口只做薄指针或镜像；`.agents` 内部说明是否避免本机绝对路径绑死
4. 仓库卫生：运行产物、缓存、截图、导出文件、`node_modules/`、`*.baiduyun.uploading.cfg` 不得进入提交边界；必要时更新 `.agents/.gitignore` 与完整性自检

**框架级治理影响评估**：如果变更触及上层入口、中层路由、流程主链路、配置真源、脚本执行路径、仓库边界，必须先做影响评估。

1. 影响层级：上层入口 / 中层路由与治理 / 流程主链路 / 执行与运行时
2. 影响宿主：Codex 桌面 / Codex CLI / Claude Code / 团队分享包
3. 影响对象：默认入口、默认路由、默认流程顺序、脚本路径、MCP/凭证、仓库卫生、历史案例复用
4. 验证方式：完整性自检、入口同步、自检脚本、最小 CLI 命令、必要的 grep 扫描

**显著修改**：仅当满足以下任一条件时才记录 `migration-log.md` → 相关 adapter/runtime-configs 检查

1. 修改的是治理规则本身（`governance/`）
2. 同时改动了多个层级的真源（如 `knowledge/` + `governance/`）
3. 这次修改会改变后续默认路由、默认判断顺序或团队统一口径
4. 需要为后续追溯“为什么这样改”保留治理级记录

以下情况通常**不必**单独记录 `migration-log.md`：

1. 仅在某个现有知识文档里补一条局部经验
2. 只修正文案、补示例、微调措辞
3. 不影响其他文档、路由或默认执行策略的小幅更新

**文件移入 _cleanup/**：_cleanup/README 登记 → 检查死链

**清理工作流**：不直接删除，先移入 `_cleanup/`，等人工确认后统一清理

## 9. 沉淀优先池（迁自 knowledge-priority-backlog.md）

> 本节原为独立文件 `governance/knowledge-priority-backlog.md`，2026-04-20 合并至此。

当发现经验值得长期复用时，先问：这个主题是否已在优先池？它属于策略、标准、路由还是技术方案？是否已积累到足够值得正式沉淀？

### P1：建议优先沉淀

| # | 主题 | 为什么值得 | 沉淀方向 |
|---|------|----------|---------|
| 1 | 定时任务/XXL-Job 验证路由 | 高频，能与真实数据策略联动 | 先看任务入口→执行日志→数据结果；"执行了"≠"生效了" |
| 2 | MQ/异步任务问题分析路由 | 高频，需跨日志、队列、数据联合判断 | 起手看队列/消费/日志/结果表；消息发了没消费 vs 消费了没生效 |
| 3 | 平台接口问题验证路由 | 高频，依赖平台真实返回 | 本地代码→服务日志→平台请求参数→平台返回结果→操作日志闭环 |
| 4 | 生产问题只读核查原则 | 高频、风险高、需稳定口径 | 什么情况只读够→什么情况必须转 staging→生产查询边界 |
| 5 | 回归范围判断策略 | 高频，影响测试效率 | 按改动点定范围→按风险定优先级→哪些必须连带回归 |
| 6 | 高风险验证的最小影响原则 | 高频，团队安全边界 | 单店铺/单批次/单对象→如何缩小影响→何时不应继续 |

### P2：建议逐步积累
