# Bug 分析结论证据标准

## 目标

这份文档用于统一 Bug 分析、Bug 提单、修复验证、测试结论中的证据标准。

核心目标：

- 让测试结论更可追溯
- 让 Agent 输出更稳定
- 减少“现象像 Bug，但证据不够”的争议
- 减少“有问题但描述不清、证据不足”的沟通成本

一句话原则：

> Bug 结论必须建立在可核查证据上，事实与推断必须分开表达。

---

## 一、适用范围

适用于以下场景：

- Bug 是否成立的分析结论
- Bug 提单前的证据整理
- 开发修复后的验证结论
- 回归测试中的“通过 / 不通过 / 待确认”判断
- 需要对外同步的问题分析结论

---

## 二、最低证据要求

一个 Bug 要想形成“可落地”的分析结论，至少应满足以下 3 类证据中的 2 类。

### 1. 现象证据

用于证明“问题确实发生了”。

常见形式：

- 页面表现异常
- 接口返回异常
- 日志报错
- 任务执行异常
- 数据结果异常

### 2. 路径证据

用于证明“问题是如何发生的”。

常见形式：

- 重现步骤
- 调用链路
- 执行参数
- 触发条件
- 任务入口与执行日志

### 3. 结果证据

用于证明“问题造成了什么结果”。

常见形式：

- MySQL / Mongo 数据与预期不符
- 操作日志缺失或错误
- 平台未生效 / 多执行 / 漏执行
- 页面状态前后不一致

如果只有“现象描述”，没有路径证据或结果证据，通常只能算“疑似问题”，不应直接下强结论。

---

## 三、推荐证据组合

### 组合 A：页面 / Web 类问题

推荐至少具备：

- 页面现象
- 接口返回或前端请求结果
- 数据库或结果状态核对

### 组合 B：定时任务 / 异步任务类问题

推荐至少具备：

- 执行入口或任务参数
- 服务日志 / 任务日志
- 操作日志或结果数据

### 组合 C：平台接口类问题

推荐至少具备：

- 请求参数
- 平台返回
- 本地落库 / 操作日志 / 页面结果

### 组合 D：数据类问题

推荐至少具备：

- 异常数据样例
- 查询 SQL / Mongo 结果
- 业务规则或代码依据

---

## 四、事实与推断的表达规范

这是最关键的一条。

### 1. 确认事实

确认事实必须来自可核查证据。

推荐表达：

- 已确认
- 日志显示
- SQL 查询结果显示
- Mongo 操作日志显示
- 代码中可见
- 生产 / staging 环境已查到

例如：

- 已确认 `store_acct_id=43633` 在 Mongo 操作日志中存在 `1381` 条 `oper_type=17` 的成功记录
- staging 日志显示该任务于 `2026-04-01 19:10:40` 开始执行，并于 `19:19:50` 正常结束

### 2. 推断判断

推断必须明确标记为判断，不能写成既定事实。

推荐表达：

- 初步判断
- 推测
- 从当前证据看，更可能是
- 暂未查到直接证据，但现象与…一致
- 倾向认为

例如：

- 从当前证据看，更可能是旧匹配条件过窄导致部分商品未进入标零任务
- 暂未查到正式需求文档，当前只能判断修复逻辑与问题现象一致

### 3. 证据不足

当证据不够时，要明确说“不足”，而不是硬下结论。

推荐表达：

- 当前证据不足以确认
- 暂无法直接证明
- 仍需补充日志 / SQL / 文档依据
- 只能确认现象，暂不能确认根因

---

## 五、Bug 是否成立的判断标准

### 可判定“Bug 成立”

通常满足以下条件：

- 现象明确存在
- 有可复核证据
- 与需求、代码逻辑或业务规则存在明确偏差

### 可判定“疑似 Bug / 待确认”

通常是以下情况：

- 现象存在，但证据链不完整
- 结果异常，但无法确认预期口径
- 有怀疑根因，但没有直接证据

### 不宜直接判定为 Bug

通常是以下情况：

- 只有主观感受，没有客观证据
- 预期本身不明确
- 问题只发生在脏数据或误操作前提下，且无业务规则支持

---

## 六、修复验证通过标准

不要把“代码改了”当成“验证通过”。

修复通过至少应区分三层：

### 1. 修复点通过

确认开发修改点与问题根因一致。

证据可包括：

- 代码 diff
- 提交记录
- 逻辑路径核对

### 2. 现象回归通过

确认原问题现象不再出现。

证据可包括：

- 原重现场景验证通过
- 原异常数据已恢复正常
- 原错误日志不再出现

### 3. 风险回归通过

确认本次修复没有明显带出新的高风险问题。

证据可包括：

- 关键连带场景验证
- 受影响主链路抽样检查

只有修复点通过，不等于整体验证通过。

---

## 七、常见误区

### 误区 1：把现象当根因

错误方式：

- “页面显示错了，所以肯定是 SQL 错了”

正确方式：

- 先确认现象，再补日志 / SQL / 代码证据，最后再判断根因

### 误区 2：把推断写成事实

错误方式：

- “需求就是这么规定的”

但实际上并没有找到需求文档。

正确方式：

- “当前未查到正式需求文档，现有代码和现象更支持……”

### 误区 3：把执行成功当结果正确

错误方式：

- “任务返回 200，所以功能没问题”

正确方式：

- 任务执行成功只能证明“跑了”，还要核对数据是否真的按预期变化

### 误区 4：只给结论，不给证据来源

错误方式：

- “已验证通过”

正确方式：

- 写明在哪个环境、用什么参数、查了什么日志、看到什么结果

---

## 八、推荐输出结构

以后无论是分析、提 Bug 还是做验证结论，推荐尽量按这个顺序表达：

1. 已确认事实
2. 初步判断 / 推断
3. 证据来源
4. 当前结论
5. 仍待补充项

---

## 九、简版模板

### 模板 A：Bug 分析

```text
已确认：xxx 现象存在，日志 / SQL / 操作记录已查到对应异常证据。
初步判断：当前更可能是 xxx 导致。
证据来源：服务日志、Mongo / MySQL 查询结果、代码位置。
结论：Bug 成立 / 待进一步确认。
```

### 模板 B：修复验证

```text
已核对修复代码与问题描述，修改点与原问题一致。
原问题场景已回归验证通过，相关日志 / 数据结果符合预期。
当前未发现明显连带异常，本次可判定为验证通过。
```

### 模板 C：证据不足

```text
当前已确认 xxx 现象，但证据仍不足以直接确认根因。
暂未查到需求 / 日志 / 数据侧直接依据，建议补充 xxx 后再做最终结论。
```

---

## 十、一句话原则

测试结论不是“像不像”，而是“有没有证据、证据是否足够、事实和推断有没有分开写清楚”。
