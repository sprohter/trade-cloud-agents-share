# 知识沉淀优先池

> ⚠️ 已 deprecated（2026-04-20）。内容已合并至 `governance/sedimentation.md` §9 沉淀优先池，本文件保留以防旧引用，不再维护。

## 目标

这份文档用于维护“接下来最值得沉淀的高价值主题清单”。

作用不是记录所有想法，而是帮助团队优先沉淀：

- 复用频率高的
- 节省时间明显的
- 能提升 Agent 判断质量的
- 能形成通用规则的

这是一份动态 backlog，可以持续补充、调整、下沉到具体文档。

---

## 使用方式

当一次任务结束后，如果发现某个经验值得长期复用，先不要急着直接新增文档，可以先问：

1. 这个主题是否已经在优先池里
2. 它属于策略、标准、路由还是技术方案
3. 是否已经积累到足够值得正式沉淀

如果答案明确，再决定：

- 新增正式文档
- 补充现有文档
- 先留在 backlog 继续观察

## 当前优先级

### P1：建议优先沉淀

#### 1. Bug 分析结论证据标准

为什么值得：

- 高频
- 直接影响测试结论质量
- 能明显提升 Agent 输出稳定性

建议沉淀方向：

- 什么叫“已确认事实”
- 什么叫“推断判断”
- 什么证据足以支撑 bug 成立
- 什么证据足以支撑“修复通过”

#### 2. 修复验证通过标准

为什么值得：

- 高频
- 经常影响开发、测试、产品之间的对齐
- 适合形成固定表达方式

建议沉淀方向：

- 修复点验证
- 回归点验证
- 风险点验证
- 结论怎么写更清晰

#### 3. Web 功能异常排查路由

为什么值得：

- 高频
- 适用范围大
- 可以快速指导 Agent 起手顺序

建议沉淀方向：

- 先看页面现象还是先抓接口
- 什么时候查日志
- 什么时候查数据库
- 什么时候转向真实数据验证

#### 4. 定时任务 / XXL-Job 验证路由

为什么值得：

- 高频
- 适合和现有真实数据策略联动
- 能减少很多重复分析

建议沉淀方向：

- 先看任务入口
- 再看执行日志
- 再看数据结果
- 如何判断“执行了”和“生效了”是两回事

#### 5. MQ / 异步任务问题分析路由

为什么值得：

- 高频
- 经常需要跨日志、队列、数据联合判断
- Agent 很适合按流程协助

建议沉淀方向：

- 起手先看队列、消费、日志还是结果表
- 如何判断消息发了但没消费
- 如何判断消费了但没生效

#### 6. 平台接口问题验证路由

为什么值得：

- 高频
- 很多问题依赖平台真实返回
- 适合和真实数据场景联动

建议沉淀方向：

- 本地代码
- 服务日志
- 平台请求参数
- 平台返回结果
- 操作日志闭环

#### 7. 生产问题只读核查原则

为什么值得：

- 高频
- 风险高
- 需要稳定口径

建议沉淀方向：

- 什么情况下只读核查就够
- 什么情况下必须转 staging / 测试验证
- 生产查询的边界和表达方式

#### 8. 测试结论中“事实 / 推断”表达规范

为什么值得：

- 高频
- 对外沟通价值高
- 能直接提升 Agent 输出质量

建议沉淀方向：

- 如何区分“确认”和“怀疑”
- 如何写“当前证据不足”
- 如何避免把推断写成事实

#### 9. 回归范围判断策略

为什么值得：

- 高频
- 经常影响测试效率
- 最能体现经验价值

建议沉淀方向：

- 按改动点定回归范围
- 按风险定回归优先级
- 哪些场景必须做连带回归

#### 10. 高风险验证的最小影响原则

为什么值得：

- 高频
- 能约束真实数据场景验证
- 是团队协作的重要安全边界

建议沉淀方向：

- 单店铺、单批次、单对象验证原则
- 如何缩小影响范围
- 什么时候不应继续执行

## 推荐维护方式

这份优先池建议这样维护：

1. 遇到高价值经验时，先看是否能挂到现有主题下
2. 如果某一主题连续出现 2-3 次，就优先正式沉淀
3. 已经正式沉淀的主题，可以从 backlog 中移除，或标注“已落地”

## 当前已落地主题

- 真实数据场景验证策略
- Bug 分析结论证据标准
- 修复验证通过标准
- Web 功能异常排查路由

## 一句话原则

优先沉淀那些“以后还会反复遇到、能明显节省时间、还能帮助 Agent 做更好判断”的主题，而不是把每次任务都写成文档。
