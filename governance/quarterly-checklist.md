# Agent 季度治理 Checklist

> ⚠️ 已 deprecated（2026-04-20）。内容已合并至 `governance/asset-registry.md` 季度治理体检节，本文件保留以防旧引用，不再维护。

> 状态：`Trial`
> Owner：`<maintainer> + Codex`
> 角色：`治理清单`

## 1. 用途

这份清单用于定期检查当前 Agent 相关资产是否开始变乱、变重、变重复。

建议使用场景：

- 新增了一批文档后
- 做完一轮较大的试运行后
- 准备做团队分享前
- 觉得“最近文档越来越多，有点乱”时

不要求必须按季度严格执行。

只要你觉得“该体检一下了”，就可以拿它来跑。

## 2. 最简单的使用方式

直接对 Agent 说：

```text
按季度治理清单检查一下当前 Agent 资产
```

如果你只想先看建议，不想改文档：

```text
按季度治理清单检查一下当前 Agent 资产，只给建议，不落库
```

## 3. 标准检查项

### 3.1 入口检查

- 当前是否已经有多个同主题主入口
- 是否有人第一次看会不知道先看哪份
- 是否有长文档还在承担入口职责

如果有问题，建议动作：

- 保留一个主入口
- 其余降级为详细说明 / 分享稿 / 背景参考

### 3.2 状态检查

- 核心文档是否已有状态标签
- `Draft / Trial / Stable / Reference / Archive` 是否合理
- 有没有文档长期处于 Trial 但其实已经稳定

如果有问题，建议动作：

- 补状态
- 调整状态

### 3.3 重叠检查

- 是否有两份以上文档持续讲同一主题
- 它们是“主文档 + 补充文档”关系，还是重复关系
- 是否存在不同入口都在重复解释同一件事

如果有问题，建议动作：

- 整合
- 减少平行文档

### 3.4 长文档检查

- 是否有超长文档还在承担主入口职责
- 是否有长文档引用很低，但仍被默认认为重要

建议参考：

- `> 300` 行：默认不作为一线入口
- `> 500` 行：更适合做背景参考

### 3.5 低价值资产检查

- 是否存在长期低引用、低复用文档
- 是否存在只适合保留历史、不适合作为现行资产的文档
- 是否存在已经完成使命的分享稿

如果有问题，建议动作：

- 降级为 `Reference`
- 或列入归档候选

### 3.6 archive 巡检

**目的**：防止有价值的历史内容永远沉底。

执行方式——对 agent 说：

```text
扫一下 .agents/archive/，找出有价值但还没进 knowledge/ 的内容，给建议清单
```

判断标准：

| 情况 | 动作 |
|------|------|
| archive 里有内容，knowledge/ 里没有对应知识 | 提取合并进 knowledge/ |
| archive 里有内容，knowledge/ 里已有覆盖 | 保持归档，不重复 |
| archive 文件超过 2 年没被引用 | 标记为可删除候选 |

**framework-design/ 和 tool-configs/ 不需要巡检**，这两批是合理遗忘的历史记录。重点巡检：`session-records/`、`ops-packages/`。

### 3.7 沉淀质量检查

- 最近新增沉淀里，有没有明显重复内容
- 有没有证据不足却进入正式知识的内容
- 有没有新结论可能和旧规则冲突

如果有问题，建议动作：

- 退回候选层
- 标注待确认
- 不直接覆盖旧知识

## 4. 输出建议格式

建议 Agent 每次按这份清单检查后，固定输出：

```text
本次治理检查结果

1. 主入口资产
2. 详细规则资产
3. 轻量辅助资产
4. 背景参考资产
5. 待整合资产
6. 待降级资产
7. 归档候选
8. 本次建议动作（P1 / P2 / P3）
```

## 5. 当前最推荐的执行节奏

现阶段更建议：

- 不强制定期
- 按阶段触发
- 一次只改少量高价值项

也就是：

- 少量治理
- 连续优化
- 不搞大规模一次性清库

## 6. 当前一句话原则

治理的目标不是“文档变少”，而是：

**让主入口更清晰，让重复更少，让有效资产更容易被用到。**
