# 钉钉测试周评审列表模板

## 适用场景

用于每周五各小组任务评审结束后，复核测试组员下周或当周任务安排是否合理。

## 默认评审口径

1. `测试难度`
   - 代表周计划评审阶段的测试工时评估
   - 数字通常表示评估工时
   - 这是评估值，不等于实际消耗工时

2. `双口径原则`
   - 周评审默认同时输出 `看板口径` 和 `实际口径`
   - `看板口径`：按 `测试` 字段和未完成状态直接统计
   - `实际口径`：再结合 `备注` 修正
   - 典型修正规则：
     - `开发全栈 / 合并开发 / 一人完成 / 已移交` → 不计入测试执行负荷
     - `技术评审 / 写文档 / 单独过测试用例` → 按 `0.5` 个任务当量计入
     - `长期任务 / 先不排期 / 暂不排期` → 不计入本周负荷
   - 如果已有人工确认的临时拆分，结论里要单独说明“按最新分工调整后”的结果

3. `测试计划完成时间`
   - 用于判断排期是否合理
   - 不能只看单个任务，需要结合成员总盘子一起判断

4. 时间链路判断
   - 逾期判断不能只看 `测试计划完成时间`
   - 需要按这条线性链路一起判断：
     - `开始时间`
     - `后端计划完成时间`
     - `前端计划完成时间`
     - `测试计划完成时间`
     - `截止时间`
   - 如果前置节点已经偏移，后续节点即使还没到日期，也可能已经存在延期风险
   - `测试难度` 需要和提测时间匹配
   - 提测时间可理解为 `后端或前端最后完成时间`

5. 每人合理工时
   - 每天 `6h` 为上限
   - 每天 `4-5h` 为中等合理水平
   - 需要预留时间给：
     - bug 修复回归
     - 业务反馈问题处理
     - 临时插入事项

6. 周评审核心问题
   - 测试难度评估是否偏低或偏高
   - 测试计划完成时间是否过紧
   - 每个人的周任务量是否合理
   - 是否已经出现堆积或潜在延期

## 默认输出形式

以后周汇报、周评审、工作量评估，优先使用列表形式输出，不优先使用大段文字。

## 列表输出结构

1. `工作量统计列表`
   - 每人的看板总任务数
   - 每人的实际任务当量
   - 每人的本周到期任务数
   - 每人的本周实际任务当量
   - 明确逾期数
   - 无日期数
   - 链路异常数

2. `风险列表`
   - 高负载成员
   - 看板虚高成员
   - 本周到期过满成员
   - 测试计划偏紧任务
   - 时间链路已偏移的任务
   - 已有逾期的任务

3. `建议列表`
   - 建议拆分的任务
   - 建议顺延的任务
   - 建议重新评估测试难度的任务
   - 需要组长介入的事项

## 周评审列表模板

### 1. 整体结论

1. 本周测试组看板总任务数：`X`
2. 本周测试组实际任务当量：`X`
3. 本周到期任务数：`X`
4. 当前最高风险成员：`A / B / C`

### 2. 每人工作量统计

1. `姓名`
   - 看板总任务数：`X`
   - 实际任务当量：`X`
   - 本周到期数：`X`
   - 本周实际任务当量：`X`
   - 明确逾期数：`X`
   - 无日期数：`X`
   - 链路异常数：`X`
   - 结论：`偏高 / 合理 / 偏低`

### 3. 每人风险判断

1. `姓名`
   - 风险点：`例如本周任务过满 / 有逾期 / 长期任务挤占 / 测试计划偏紧`
   - 建议：`例如拆分 / 顺延 / 调整优先级 / 重新评估测试难度`

### 4. 需要组长介入的事项

1. `成员 + 任务`
   - 原因：`为什么需要介入`
   - 建议动作：`怎么调整`
